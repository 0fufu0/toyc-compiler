# ToyC 编译器设计与实现实训报告

## 摘 要

本实验旨在设计并实现一个面向 ToyC 语言的完整编译器，将 ToyC 源程序编译为 RISC-V32 汇编代码。实验遵循软件工程模块化设计原则，将编译器划分为前端（词法/语法分析）、中端（语义分析/IR 生成）和后端（代码生成）三个主要阶段。

实验设计主要遵循以下原则：采用 ANTLR4 进行语法分析，使用 Visitor 模式构建自定义 AST，通过三地址码作为中间表示解耦中端与后端，采用快速失败机制处理错误。

实验内容主要包括：设计 ToyC 语言文法并实现词法/语法分析器；实现语义分析器进行类型检查、作用域管理和常量折叠；将 AST 转换为三地址码形式的中间表示；最后将中间表示翻译为 RISC-V32 汇编代码。

实验结论为：本项目成功实现了一个功能完整的 ToyC 编译器，支持变量声明、常量定义、算术/逻辑运算、条件分支、循环结构、函数调用及递归等核心语言特性，能够生成正确可执行的 RISC-V32 汇编代码。

**关键词**：编译器；ToyC；RISC-V32；ANTLR4；三地址码

## 1. 实验目的和意义

### 1.1 实验目的

本实验的主要目的是设计并实现一个面向 ToyC 语言的编译器。ToyC 是 C 语言的一个简化子集，省去了数组、指针、I/O、多文件编译等特性，适合作为编译器教学的实践载体。

通过本实验，我们期望达成以下具体目标：

1. **理解编译器架构**：通过实现一个完整的编译器，深入理解编译器的前端（词法分析、语法分析）、中端（语义分析、中间代码生成）和后端（代码优化、目标代码生成）的基本原理和相互关系。

2. **掌握语法分析技术**：学习使用 ANTLR4 词法/语法分析器生成工具，理解上下文无关文法在实际语言实现中的应用，掌握从文法规则到可执行编译器的转换过程。

3. **理解语义分析机制**：实现符号表的构造与管理，理解类型检查、作用域规则、常量折叠等语义分析的核心概念。

4. **掌握代码生成技术**：将中间表示翻译为 RISC-V32 目标代码，理解不同体系结构下的指令选择、寄存器分配和栈帧布局等底层问题。

5. **培养工程协作能力**：通过小组分工协作，理解大型软件项目中模块划分、接口定义和集成测试的重要性。

### 1.2 实验意义

编译器是计算机科学中的核心基础软件，其设计理念和实现技术广泛应用于各类软件工程领域。本实验的意义体现在以下几个方面：

**理论联系实际**：编译器原理是计算机专业的重要课程内容，但往往停留在理论层面。通过本实验，学生能够将词法分析、语法分析、语义分析、代码生成等理论知识应用于实际工程，加深对课堂内容的理解。

**体系结构知识巩固**：将高级语言翻译为 RISC-V32 汇编代码的过程，要求学生深入理解目标体系的指令集、寄存器约定、调用规范等硬件特性，从而巩固计算机体系结构相关知识。

**软件工程能力提升**：本项目采用模块化设计，各模块之间通过清晰定义的接口协作。这种设计方式训练了学生的抽象思维能力和大型软件开发能力。

**为系统软件研发奠定基础**：编译器的设计与实现能力是从事系统软件（如运行时环境、编程语言设计、工具链开发等）研发的重要基础，本实验为未来深入学习打下坚实基础。

## 2. 实验设计

### 2.1 概述

本编译器采用经典的三阶段架构：前端负责将源程序文本转换为抽象语法树（AST）；中端负责语义检查和生成中间表示（IR）；后端负责将中间表示转换为目标机器代码。

整个项目的技术选型如下：

- **开发语言**：Java（JDK 21）
- **构建工具**：Maven
- **语法分析器生成工具**：ANTLR 4.13.1
- **目标体系结构**：RISC-V32

编译器各模块的职责划分和接口约定遵循团队契约规范（参见接口说明.md），采用 Visitor 模式实现 AST 遍历，使用三地址码作为中端与后端的接口约定。

### 2.2 前端设计

#### 2.2.1 词法/语法分析

前端使用 ANTLR4 根据 ToyC 语言文法自动生成词法分析器和语法分析器。ToyC 语言的文法定义遵循以下核心规则：

```
compUnit    : (decl | funcDef)+ EOF
decl        : constDecl | varDecl
constDecl   : CONST INT ID ASSIGN expr SEMI
varDecl     : INT ID ASSIGN expr SEMI
funcDef     : (INT | VOID) ID LPAREN paramList? RPAREN block
expr        : lOrExpr
lOrExpr     : lAndExpr | lOrExpr OR lAndExpr
lAndExpr    : relExpr | lAndExpr AND relExpr
relExpr     : addExpr | relExpr (LT | GT | LE | GE | EQ | NE) addExpr
addExpr     : mulExpr | addExpr (PLUS | MINUS) mulExpr
mulExpr     : unaryExpr | mulExpr (STAR | SLASH | MOD) unaryExpr
unaryExpr   : primaryExpr | (PLUS | MINUS | NOT) unaryExpr
primaryExpr : ID | NUMBER | LPAREN expr RPAREN | ID LPAREN argList? RPAREN
```

词法规则方面，标识符 ID 的正则表达式为 `[_A-Za-z][_A-Za-z0-9]*`，整数 NUMBER 的正则表达式为 `-?(0|[1-9][0-9]*)`。空白字符和注释（单行注释 `//` 和多行注释 `/* */`）被忽略。

#### 2.2.2 AST 设计与构建

前端不直接暴露 ANTLR 生成的 ParseTree，而是通过 `AstBuilder` 类将其转换为自定义的 AST 结构。这种设计使得 AST 节点可以包含额外的语义信息，且对中端和后端隐藏了 ANTLR 的内部实现细节。

AST 节点类均继承自抽象基类 `AstNode`，采用 Visitor 模式实现双分派：

```java
public abstract class AstNode {
    public int line = -1;
    public int column = -1;
    public abstract <T> T accept(AstVisitor<T> visitor);
}
```

各节点类型定义如下：

| 节点类型 | 说明 |
|---------|------|
| CompUnitNode | 编译单元根节点 |
| FuncDefNode | 函数定义节点 |
| ParamNode | 形参节点 |
| BlockStmtNode | 语句块节点 |
| IfStmtNode | if-else 条件分支节点 |
| WhileStmtNode | while 循环节点 |
| BreakStmtNode / ContinueStmtNode | 循环控制节点 |
| ReturnStmtNode | return 语句节点 |
| AssignStmtNode | 赋值语句节点 |
| ExprStmtNode | 表达式语句节点 |
| EmptyStmtNode | 空语句节点 |
| ConstDeclNode / VarDeclNode | 常量/变量声明节点 |
| BinaryExprNode | 二元表达式节点 |
| UnaryExprNode | 一元表达式节点 |
| CallExprNode | 函数调用节点 |
| IdNode | 标识符引用节点 |
| IntLiteralNode | 整数字面量节点 |

每个节点均包含源码位置信息（line 和 column），便于调试和错误报告。

#### 2.2.3 前端入口

`ToyCFrontend` 类是前端模块的入口点，提供静态方法 `parse(String source)` 将源代码字符串解析为 `CompUnitNode`（AST 根节点）。该类使用 ANTLR4 的 `CharStreams`、`CommonTokenStream` 和 `CommonTokenStream` 构建词法/语法分析器，并配置自定义的错误监听器以在遇到词法或语法错误时抛出包含行列号的异常。

### 2.3 中端设计

#### 2.3.1 语义分析

语义分析器 `SemanticAnalyzer` 实现 `AstVisitor<Void>` 接口，在 AST 遍历过程中完成以下任务：

1. **符号表管理**：维护一个嵌套的作用域结构（通过 `SymbolTable` 类实现），在进入新作用域时创建新层，在离开时恢复。符号表项包含变量/常量的名称、类型、是否为常量等信息。

2. **重复定义检查**：在声明新的符号时，检查当前作用域是否已存在同名符号。若存在则报错。

3. **未定义引用检查**：在引用标识符时，检查符号表中是否存在对应的定义。若未找到则报错。

4. **类型检查**：验证赋值语句右值类型与左值类型匹配；验证函数调用的参数个数和类型；验证 return 表达式类型与函数返回类型一致。

5. **const 约束检查**：确保 const 修饰的标识符不能作为赋值语句的左值。

6. **循环控制语句检查**：确保 break 和 continue 语句仅出现在循环语句（while）内部。

7. **函数返回检查**：确保返回类型为 int 的函数在所有可能的执行路径上都有 return 语句返回值。

8. **符号引用填充**：为每个 IdNode 填写其对应的 Symbol 对象引用（`symbolRef` 字段），供后端使用。

9. **常量折叠**：对于编译期即可确定值的常量表达式（如字面量运算、常量与常量运算），计算其数值并填入节点的 `constValue` 字段。

#### 2.3.2 中间表示生成

IR 生成器 `IrGenerator` 实现 `AstVisitor<IrList>` 接口，将经过语义分析的 AST 转换为三地址码形式的中间表示。

三地址码指令使用 `IrInst` 类表示，每条指令包含操作码（op）和最多三个操作数（dst、a、b）。支持的操作码包括：

| 操作码 | 说明 | 示例 |
|-------|------|------|
| CONST | 加载立即数 | `CONST t0, 10` |
| ASSIGN | 变量赋值 | `ASSIGN a, t0` |
| BIN_ADD/SUB/MUL/DIV/MOD | 算术运算 | `BIN_ADD t2, t0, t1` |
| BIN_LT/GT/LE/GE/EQ/NE | 关系运算 | `BIN_LT t2, t0, t1` |
| LABEL | 定义标签 | `LABEL L1:` |
| GOTO | 无条件跳转 | `GOTO L1` |
| IFZ/IFNZ | 条件跳转 | `IFZ t0, L1` |
| CALL | 函数调用 | `CALL fib` |
| RET | 函数返回 | `RET t0` |
| FUNC/ENDFUNC | 函数边界 | `FUNC main:` |

对于短路求值逻辑与（`&&`）和逻辑或（`||`）表达式，IR 生成器通过条件跳转指令实现，避免对右侧操作数进行不必要的求值。

`IrList` 类是 IR 指令列表的容器，提供 `newTemp()` 和 `newLabel()` 方法用于生成临时变量名和标签名。

### 2.4 后端设计

#### 2.4.1 代码生成架构

后端代码生成器 `CodeGenerator` 接受 `IrList`（三地址码指令列表），输出 RISC-V32 汇编代码文本。代码生成过程分为以下阶段：

1. **指令发射**：遍历 IR 指令列表，根据操作码分发到对应的生成方法。

2. **寄存器分配（简单策略）**：采用基于栈的分配策略，将所有变量存储在栈帧中。函数入口处计算所有变量的偏移量，分配栈空间（16 字节对齐）。

3. **栈帧布局**：每个函数调用分配一个栈帧，包含局部变量区域和返回地址（ra）存储位置。

4. **指令生成**：为每种 IR 指令生成对应的 RISC-V 汇编指令。

#### 2.4.2 RISC-V32 指令选择

各操作码对应的 RISC-V 指令映射如下：

| IR 操作码 | RISC-V 指令 |
|----------|------------|
| CONST | `li rd, imm` |
| ASSIGN | `lw rs, offset(sp)` / `sw rs, offset(sp)` |
| BIN_ADD | `add rd, rs1, rs2` |
| BIN_SUB | `sub rd, rs1, rs2` |
| BIN_MUL | `mul rd, rs1, rs2` |
| BIN_DIV | `div rd, rs1, rs2` |
| BIN_MOD | `rem rd, rs1, rs2` |
| BIN_LT | `slt rd, rs1, rs2` |
| BIN_GT | `slt rd, rs2, rs1` |
| BIN_LE | `slt rd, rs2, rs1` + `xori` |
| BIN_GE | `slt rd, rs1, rs2` + `xori` |
| BIN_EQ | `sub` + `seqz` |
| BIN_NE | `sub` + `snez` |
| IFZ | `beqz rs, label` |
| IFNZ | `bnez rs, label` |
| GOTO | `j label` |
| CALL | `sw ra, offset(sp)` / `call func` / `lw ra, offset(sp)` |
| RET | `addi sp, sp, framesize` / `ret` |

#### 2.4.3 程序入口处理

生成的汇编代码以 `_start` 作为程序入口点，该入口点调用 `main` 函数并通过 `ecall` 指令退出。这种处理方式符合 RISC-V Linux 程序的标准约定。

## 3. 实验结论

本项目成功实现了一个功能完整的 ToyC 语言编译器，能够将符合 ToyC 语言规范的源程序编译为可执行的 RISC-V32 汇编代码。

**主要成果**：

1. **前端实现**：使用 ANTLR4 生成了词法/语法分析器，实现了从源代码文本到抽象语法树的转换。AST 采用 Visitor 模式设计，支持中端和后端的解耦访问。

2. **中端实现**：完成了语义分析器的设计与实现，包括符号表管理、类型检查、作用域检查、常量折叠等功能。实现了从 AST 到三地址码中间表示的转换，支持短路求值等语义的正确实现。

3. **后端实现**：完成了 RISC-V32 代码生成器的设计与实现，采用基于栈的变量分配策略，生成符合 RISC-V ABI 约定的汇编代码。

**支持的语言特性**：

- 变量声明与赋值
- 常量声明与使用
- 算术运算（加、减、乘、除、取模）
- 关系运算（小于、大于、小于等于、大于等于、等于、不等于）
- 逻辑运算（短路求值的逻辑与、逻辑或）
- 一元运算（正号、负号、逻辑非）
- 条件分支（if-else）
- 循环结构（while）
- 循环控制（break、continue）
- 函数定义与调用
- 函数递归
- 嵌套作用域

**已知局限性**：

- 尚未实现代码优化Pass（如常量传播、死代码消除、公共子表达式消除等）
- 寄存器分配采用简单的基于栈的策略，未实现图着色等高级寄存器分配算法
- 未支持数组、指针、多文件编译等 C 语言特性
