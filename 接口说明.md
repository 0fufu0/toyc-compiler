#### 📋 表 1：AST 架構與節點定義 (A 與全員的契約)

**目的：統一資料結構，避免 B 和 C 無法在節點上附加資料。**

|**模塊 / 職責**|**負責角色**|**標準規範約定**|**Java 具體示例 (A 需定義)**|
|---|---|---|---|
|**AST 基底類別**|A (Parser)|不直接暴露 ANTLR 樹，由 A 定義**自定義的 OOP 類別**。|`abstract class AstNode {}`<br><br>  <br><br>`abstract class ExprNode extends AstNode {}`|
|**節點命名**|A (Parser)|統一採用 **PascalCase + Node 後綴**。|`VarDeclNode`, `IfStmtNode`, `BinaryExprNode`|
|**預留擴充欄位**|A (Parser)|AST 節點需開放權限，允許 B 在語義分析後填寫附加資料。|在特定節點預留：<br><br>  <br><br>`public Symbol symbolRef;`<br><br>  <br><br>`public Integer constValue;`|

#### 📋 表 2：AST 遍歷機制 (A 提供介面，B 與 C 實作)

**目的：讓語義分析和 IR 生成完全解耦，互不干涉。**

|**遍歷階段**|**負責角色**|**實現方式約定**|**備註 / 行動指南**|
|---|---|---|---|
|**定義介面**|A (Parser)|提供 **Visitor 模式**介面 `interface AstVisitor<T>`|這是標準寫法，A 只需定義好 `visit` 方法，不參與後續邏輯。|
|**語義分析**|B (語義)|實作 `class SemanticVisitor implements AstVisitor<Void>`|B 負責檢查類型、建構符號表，並在節點填上資料。|
|**IR 生成**|C (中端)|實作 `class IrVisitor implements AstVisitor<IrList>`|C 直接讀取 B 處理完畢的 AST，生成中間代碼。|

#### 📋 表 3：中端核心介面 —— 符號與常量 (B 提供給 C 的承諾)

**目的：明確 C 可以從 B 這裡拿到什麼現成的資料，減少 C 的工作量。**

|**數據類型**|**B 的處理承諾 (我來做)**|**C 的使用方式 (你直接用)**|**AST 欄位定義要求**|
|---|---|---|---|
|**符號引用**<br><br>  <br><br>(變數是誰)|遍歷聲明時建表，在所有 `IdNode` (標識符節點) 填寫對應的 Symbol 物件。|**C 無需自己查表！**<br><br>  <br><br>直接讀取節點中的 `symbolRef` 獲取作用域和類型。|在 `IdNode` 中：<br><br>  <br><br>`public Symbol symbolRef;`|
|**常量折疊**<br><br>  <br><br>(是否為常數)|識別 `const` 且在編譯期計算出所有常量表達式的數值。|**C 無需生成運算指令！**<br><br>  <br><br>若 `constValue` 不為 null，直接生成立即數賦值。|在 `ExprNode` 中：<br><br>  <br><br>`public Integer constValue = null;`|
|**類型檢查**|檢查所有賦值、函數調用和返回值類型。|**C 無需考慮類型錯誤！**<br><br>  <br><br>到達 C 的 AST 保證 100% 語義正確。|(遇錯直接拋出異常)|

#### 📋 表 4：後端 IR 與匯編介面 (C 與 D 的生死契約)

**目的：決定專案能否跑通，D 必須能看懂 C 生成的東西。**

|**規範項目**|**負責角色**|**約定標準**|**避免踩坑說明**|
|---|---|---|---|
|**IR 數據結構**|C (中端)|採用**三位址碼的物件列表** (如 `List<IrInst>`)。|**嚴禁只傳遞純字串！** D 需要遍歷物件屬性（op, arg1, result）來做暫存器分配。|
|**短路求值**|C (中端)|遇到 `&&`、`||
|**匯編輸出**|D (後端)|將 IR 翻譯為 RISC-V32 文本，輸出至 `stdout`。|確保支援 Linux 標準輸入輸出重定向，否則線上評測系統會抓不到結果。|
