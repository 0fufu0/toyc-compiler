#### 📋 表 1：AST 架构与节点定义 (A 与全员的契约)

**目的：统一数据结构，避免 B 和 C 无法在节点上附加数据。**

|**模块 / 职责**|**负责角色**|**标准规范约定**|**Java 具体示例 (A 需定义)**|
|---|---|---|---|
|**AST 基底类别**|A (Parser)|不直接暴露 ANTLR 树，由 A 定义**自定义的 OOP 类别**。|`abstract class AstNode {}`<br><br>  <br><br>`abstract class ExprNode extends AstNode {}`|
|**节点命名**|A (Parser)|统一采用 **PascalCase + Node 后缀**。|`VarDeclNode`, `IfStmtNode`, `BinaryExprNode`|
|**预留扩充字段**|A (Parser)|AST 节点需开放权限，允许 B 在语义分析后填写附加数据。|在特定节点预留：<br><br>  <br><br>`public Symbol symbolRef;`<br><br>  <br><br>`public Integer constValue;`|

#### 📋 表 2：AST 遍历机制 (A 提供接口，B 与 C 实现)

**目的：让语义分析和 IR 生成完全解耦，互不干涉。**

|**遍历阶段**|**负责角色**|**实现方式约定**|**备注 / 行动指南**|
|---|---|---|---|
|**定义接口**|A (Parser)|提供 **Visitor 模式**接口 `interface AstVisitor<T>`|这是标准写法，A 只需定义好 `visit` 方法，不参与后续逻辑。|
|**语义分析**|B (语义)|实现 `class SemanticVisitor implements AstVisitor<Void>`|B 负责检查类型、构建符号表，并在节点填上数据。|
|**IR 生成**|C (中端)|实现 `class IrVisitor implements AstVisitor<IrList>`|C 直接读取 B 处理完毕的 AST，生成中间代码。|

#### 📋 表 3：中端核心接口 —— 符号与常量 (B 提供给 C 的承诺)

**目的：明确 C 可以从 B 这里拿到什么现成的数据，减少 C 的工作量。**

|**数据类型**|**B 的处理承诺 (我来做)**|**C 的使用方式 (你直接用)**|**AST 字段定义要求**|
|---|---|---|---|
|**符号引用**<br><br>  <br><br>(变量是谁)|遍历声明时建表，在所有 `IdNode` (标识符节点) 填写对应的 Symbol 对象。|**C 无需自己查表！**<br><br>  <br><br>直接读取节点中的 `symbolRef` 获取作用域和类型。|在 `IdNode` 中：<br><br>  <br><br>`public Symbol symbolRef;`|
|**常量折叠**<br><br>  <br><br>(是否为常数)|识别 `const` 且在编译期计算出所有常量表达式的数值。|**C 无需生成运算指令！**<br><br>  <br><br>若 `constValue` 不为 null，直接生成立即数赋值。|在 `ExprNode` 中：<br><br>  <br><br>`public Integer constValue = null;`|
|**类型检查**|检查所有赋值、函数调用和返回值类型。|**C 无需考虑类型错误！**<br><br>  <br><br>到达 C 的 AST 保证 100% 语义正确。|(遇错直接抛出异常)|

#### 📋 表 4：后端 IR 与汇编接口 (C 与 D 的生死契约)

**目的：决定项目能否跑通，D 必须能看懂 C 生成的东西。**

|**规范项目**|**负责角色**|**约定标准**|**避免踩坑说明**|
|---|---|---|---|
|**IR 数据结构**|C (中端)|采用**三地址码的对象列表** (如 `List<IrInst>`)。|**严禁只传递纯字符串！** D 需要遍历对象属性（op, arg1, result）来做寄存器分配。|
|**短路求值**|C (中端)|遇到 `&&`、`||` 时生成带条件跳转的 IR（短路求值）。|功能测试必考，禁止 eager 求值两侧。|
|**汇编输出**|D (后端)|将 IR 翻译为 RISC-V32 文本，输出至 `stdout`。|确保支持 Linux 标准输入输出重定向，否则线上评测系统会抓不到结果。|

#### 📋 表 5：团队工程与防呆规范 (全员共通)

**目的：7 天极限开发，一切以「跑通」和「不冲突」为最高指导原则。**

| **规范类别**     | **具体标准约定**                                                         | **执行方式**                                    |
| ------------ | ------------------------------------------------------------------ | ------------------------------------------- |
| **错误处理机制**   | 采用 **Fail-fast (快速失败)**。遇错直接 `throw new SemanticException("...")`。 | 不浪费时间写错误恢复（Error Recovery），只要保证正确的程序能编译过即可。 |
| **Git 分支策略** | 严禁直推 `main`。每人建专属分支：`dev-a`, `dev-b`, `dev-c`, `dev-d`。            | 每天晚上合并一次，合并前必须确保本地能跑通基本的 Test Case。         |
